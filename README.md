  * обязательно покрыть:
    * успешную загрузку данных
    * ошибку загрузки
    * retry() после ошибки
    * корректное начальное состояние экрана
  * минимум 2 юнит-теста на слой данных / бизнес-логику, например:
      * отсутствие дублей в избранном / истории / кеше
      * корректное преобразование моделей
      * корректная обработка пустого результата
      * корректная обработка SavedStateHandle
*  минимум 3 интеграционных теста
  * Хотя бы 1 тест на интеграцию data-слоя:
      * Repository + Fake API + Room
  * Хотя бы 1 тест на интеграцию UI:
      * список -> клик по элементу -> переход на детали
      * ошибка -> нажатие Retry -> успешное состояние
      * отображение корректного состояния экрана после загрузки данных
* минимум 1 тест на полную последовательность эмиссий
* минимум 1 тест на нетривиальное потоковое поведение

## Выбранный API - spoonacular API: https://spoonacular.com/food-api/docs
**Были использованы следующие API эндпоинты:**

* GET https://api.spoonacular.com/recipes/complexSearch?query=  - для поиска блюд по названию
* GET https://api.spoonacular.com/recipes/{id}/information - получение ингридиентов для конкретного блюда по id
* GET https://api.spoonacular.com/recipes/{id}/analyzedInstructions - получение шагов для приготовления блюда по id

Доступ к API предоставляется по ключу, который можно получить после регистрации на сайте https://spoonacular.com/food-api/console

Количество запросов ограничено 50 в день для бесплатного пользования😢😭

## Unit-тесты:
1. Инициализация и начальное состояние
    * Проверка, что начальное состояние uiState равно RecipeBookUiState().
2. Поиск (search)
    * Успешный поиск с debounce – состояние обновляется корректно.
    * Ошибка при поиске – выставляется searchError, список результатов очищается.
    * Пустой результат поиска – список пуст, ошибки нет.
    * Повторный поиск после ошибки (retrySearch) – успешно выполняется и восстанавливает состояние.
    * Отмена предыдущего поиска при быстрой смене запроса – выполняется только последний запрос.

3. Кэшированные результаты
    * При пустом поисковом запросе (query.isBlank()) загружаются кэшированные данные (getLastCachedMeals).
    * Успешная загрузка кэша при инициализации – флаг isCacheLoading меняется с true на false, результаты отображаются.

4. История (history)
   * Успешная загрузка истории при создании ViewModel.
   * Ошибка загрузки истории – выставляется historyError.

5. Загрузка рецепта (loadRecipe)
    * Успешная загрузка рецепта – обновляется selectedRecipe, рецепт добавляется в историю, вызывается обновление истории, флаги загрузки и ошибки сбрасываются.
      
6. Корректное преобразование моделей

* Из них Нетривиальные тесты:
    * retry after error performs search again and succeeds - Проверяет, что после ошибки можно повторить поиск, и он успешно завершается. Важно для UX.
    * updateSearchQuery with blank query loads cached results - Проверяет специальное поведение: пустая строка не должна запускать сетевой поиск, а должна показывать кэш. Плюс проверяет, что кэш может обновиться после предыдущего поиска.
    * loadRecipe adds recipe to history and updates state - Интегрирует три репозитория (meal, history, favourite не используется). Проверяет последовательность вызовов: получение рецепта → добавление в историю → обновление списка истории → обновление UI состояния.
    * new search cancels previous one - Проверяет механизм debounce и отмены корутины: при быстром вводе двух запросов выполняется только последний. Использует задержки и проверку количества вызовов репозитория.

## Интеграционные тесты
1. HistoryRepositoryIntegrationTest (интеграционные тесты репозитория истории)
    * Добавление в историю и получение списка – порядок от новых к старым (последний добавленный – первый в списке).
    * Обновление существующей записи – при повторном добавлении блюда с тем же ID обновляется его информация, а не создаётся дубликат.
    * Пустая история – getHistory() возвращает пустой список, если ничего не добавляли.
      
2. NavigationIntegrationTest (интеграционные тесты навигации между экранами)
    * Переход Search → Detail – клик по блюду открывает детальный экран с правильным mealId.
    * Переход Search → Favourites – нажатие на кнопку избранного открывает экран избранного.
    * Переход Search → History – нажатие на кнопку истории открывает экран истории.
    * Возврат назад из Favourites в Search – нажатие кнопки "Back" возвращает на предыдущий экран.
    * Переход Favourites → Detail – клик по блюду в списке избранного открывает детальный экран с правильным mealId.

3. SearchScreenTest (UI-тест экрана поиска с Hilt)
    * Успешный поиск – ввод текста в поле поиска приводит к отображению результатов (после debounce), индикатор загрузки исчезает, сообщения об ошибках и "ничего не найдено" не появляются.

* Из них нетривиальные:
  * HistoryRepositoryIntegrationTest.addToHistory_withSameId_shouldUpdateExistingEntry – проверяет логику обновления, а не простого добавления дубликата.
  * NavigationIntegrationTest.pressingBackFromFavourites_returnsToSearch – проверяет корректность навигационного стека (popBackStack).
  * SearchScreenTest.search_displaysResults_success – включает синхронизацию с debounce, ожидание исчезновения индикатора загрузки и появления результатов через waitUntil – требует учёта асинхронности и таймингов.

## Flow / StateFlow
Проверяются последовательности эмиссий:
- `[] → [A] → [A, B] → [B]` (добавление, добавление, удаление)
- `[] → [A] → [A']` (обновление без дубля)
  
Проверяемое нетривиальное потоковое поведение:
- полная последовательность эмиссий при добавлении и удалении из избранного
- повторное добавление одного и того же рецепта не создаёт дубликат в потоке


## Что хранится в Room:
1. таблица favourites_meals с полями:
   * id - идентификатор блюда
   * title - название блюда
   * image - ссылка на изображение блюда
* сценарий: избранные блюда остаются после перезапуска приложения
* как проверить: перезапустить приложение, на странице избранного останутся избранные блюда
2. таблица search_meals, такие же поля как и у favourites_meals, в таблице храняться результаты последнего поиска
* сценарий: при перезапуске приложения в списке показываются последние найденные блюда
* как проверить: перезапустить приложение, в списке вместо приглашения ввести название появятся карточки блюд
3. таблица history, те же поля + поле viewedAt - содержит время просмотра деталей блюда
* сценарий: на вкладке истории показываются карточки блюд, на детали которого переходили, отсортированы по новизне
* как проверить: перезапустить приложение, на экране истории сохраняться карточки блюд, если нажать на детали блюда, а потом перейти в историю, то это блюдо будет в верху списка
